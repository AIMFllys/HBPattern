# 项目当前状态总结

> 日期：2026-04-28
> 范围：结合 `docs/plans/MVP`、Next.js 16.2 本地文档、当前项目源码与验证命令的审查结果
> 编码要求：本文件按 UTF-8 中文内容编写

## 一句话结论

项目的 MVP 主体功能已经基本搭起来了，目录大方向也符合 Next.js App Router 项目组织方式，但当前还不能算 production-ready。核心原因是 `next build` 明确失败，并且 MVP 文档中“全部完成、零 Mock、零警告”的描述与当前代码状态不完全一致。

## 当前可确认的好状态

- `src/app`、`src/components`、`src/lib`、`src/types`、`src/data/mock` 的总体分层是合理的，符合 Next.js 16.2 文档中“路由放 app，共享代码可放 app 外”的组织方式。
- TypeScript 类型检查通过：`npx.cmd tsc --noEmit` exit 0。
- Material Symbols 图标字体已经本地化到 `public/fonts/material-symbols-outlined.woff2`，`globals.css` 中没有继续使用 `fonts.googleapis.com` 的图标 CSS。
- 首页、画廊、详情、上传、个人中心、后台审核等主链路的代码已经接入 Supabase 查询/API，不是纯静态页面。
- 文本文件扫描未发现 Unicode replacement character `�`，目前没有明显中文编码损坏迹象。

## 当前阻塞项

1. `next build` 失败，属于最高优先级。

   Next.js 16.2 明确报错：Server Component 中不允许使用 `dynamic(..., { ssr: false })`。当前命中：

   - `src/components/layout/SiteHeader.tsx`
   - `src/app/gallery/[id]/page.tsx`

   这不是网络问题，而是 Next 16.2 的真实架构约束。

2. 字体仍依赖 `next/font/google`。

   `src/app/layout.tsx` 使用 `Noto_Serif_SC`、`Noto_Sans_SC`、`Newsreader`，生产构建会请求 Google Fonts。当前环境下请求失败。MVP 报告也已经把这个列为部署前 P0。

3. Next.js 16.2 提示 `middleware` 文件约定已废弃。

   当前使用 `src/middleware.ts`，构建警告要求迁移到 `proxy` 文件约定。

4. Turbopack workspace root 推断异常。

   构建提示父目录 `C:\project\package-lock.json` 与本项目 `C:\project\HBPattern\package-lock.json` 同时存在，Next 选择了父目录作为 workspace root。需要在 `next.config.ts` 设置 `turbopack.root`，或清理父级 lockfile。

## MVP 文档与代码不一致

- `docs/plans/MVP/MVP开发完整规划.md` 写了“所有数据来自 Supabase 云端，零 Mock”。
- 当前 `src/app/map/page.tsx` 仍使用 `src/data/mock/regions.ts`，并且地图页存在硬编码统计数据。
- `docs/plans/MVP/MVP完成报告.md` 写了 `eslint src` 零错误零警告，但当前 `package.json` 的 `lint` 是 `eslint` 全量扫描，实际结果是 0 errors、1 warning。
- `scripts/seed.ts` 仍是 TODO 骨架，并引用旧的 `src/app/_mock` 路径；真正可用的是 `prisma/seed.ts`。这两个 seed 文件职责不清。

## 架构风险

- `src/types/pattern.ts` 已定义 `PatternListItem`，但 `src/lib/queries.ts` 又定义了另一个同名 `PatternListItem`，字段命名和结构不同。类型目录现在没有成为单一事实源，后续容易发生 DTO 漂移。
- API 分页类型 `PaginatedResponse` 中字段是 `pageSize`，实际 `/api/patterns` 返回的是 `limit`。这会导致前后端契约名义统一、实际不统一。
- `SiteHeader` 的拆分思路是对的，但当前实现把 `ssr:false` dynamic 放在 Server Component 内，Next 16.2 不允许。需要改成合法的 Client Wrapper，或直接让移动抽屉作为普通 Client Component 被 Server Component 引用。
- 详情页评论区同样不能在 Server Component 中直接 `dynamic(..., { ssr:false })`。如果要禁用 SSR，需要单独创建 Client 包装组件；如果不强制禁用 SSR，可直接渲染 Client Component。

- `src/app/page.tsx`：183 行。

这些不是必须立即拆，但后续维护会更舒服的方向是拆出首页 sections、地图侧栏/锚点/洞察卡、画廊筛选/分页、登录表单品牌区。

## 其他风险

- `src/app/api/upload/route.ts` 服务端只根据文件名扩展名和 `file.type` 上传，缺少严格的服务端 MIME、大小、扩展名白名单校验。
- 上传流程中，如果图片上传成功但 `/api/patterns` 创建失败，当前没有清理已上传对象。
- 登录页中“忘记密码”和“立即注册”仍是 `href="#"`，属于未完成交互。
- `prisma/seed.ts` 有未使用变量 `dyeing`，导致当前 `npm.cmd run lint` 不是零警告。

## 验证结果

- `npx.cmd tsc --noEmit`：通过。
- `npm.cmd run lint`：0 errors，1 warning。
- `npm.cmd run build`：失败。

构建失败的主要原因：

- Server Component 内使用 `dynamic(..., { ssr:false })`。
- `next/font/google` 拉取 Google Fonts 失败。
- `middleware` 约定废弃警告。
- workspace root 推断到父目录的警告。

## 建议修复优先级

1. 先修 `next build` 的真实架构错误：处理 `SiteHeader.tsx` 和详情页评论区的 `ssr:false` dynamic。
2. 把 `next/font/google` 改成本地字体，至少先保证中文字体和英文字体不依赖 Google 网络。
3. 处理 Next 16.2 的 `middleware` 到 `proxy` 迁移，并设置 `turbopack.root`。
4. 统一 lint 验收口径：要么脚本改成只扫 `src`，要么修掉 `prisma/seed.ts` warning 并保持全量 lint。
5. 清理重复/不明确文件：删除或改造 `scripts/seed.ts`，明确只保留一个 seed 入口。
6. 统一 DTO 类型：让 `src/types` 成为真实的共享契约，移除 `src/lib/queries.ts` 中重复定义。
7. 再处理 MVP 口径问题：地图页要么迁移到 Supabase，要么在 MVP 文档中明确它仍是 Mock/展示页。
