# ADR-001: 引入 Python FastAPI 作为 AI 计算微服务

## Status

**Accepted** — 2026-04-29

---

## Context

### 问题描述

HBPattern 项目需要支持垂直领域 AI 服务，包括：
1. 学术论文 / PDF 文档的结构化解析与知识入库
2. 基于私有知识库的检索增强生成（RAG）
3. 未来可能的自研算法（基于特定论文设计的独特模型）

### 约束条件

| 约束 | 详情 |
|------|------|
| **团队规模** | 个人开发者（1 人） |
| **预算** | 有限（初期无 GPU 服务器预算） |
| **部署环境** | 宝塔面板 VPS |
| **现有架构** | Next.js 16 + Prisma + Supabase，已稳定运行 |
| **时间线** | 渐进式演进，非一次性重构 |

---

## Options Considered

### Option A：保持纯 Next.js，通过第三方 API 实现所有 AI 功能

| 维度 | 评估 |
|------|------|
| **优点** | 零额外运维成本；架构简单；开发速度快 |
| **缺点** | 无法精细控制文档解析逻辑；依赖第三方 RAG API 的质量；无法复现论文算法；长期成本高（API 调用费）|
| **复杂度** | 低 |
| **适用场景** | AI 功能仅限简单对话，无垂直领域深度需求 |

### Option B：在 Next.js 中引入 Node.js AI 库（LangChain.js + Transformers.js）

| 维度 | 评估 |
|------|------|
| **优点** | 单一代码库；无跨服务通信开销 |
| **缺点** | LangChain.js 功能严重滞后于 Python 版；Transformers.js 仅支持小模型；CPU 密集任务阻塞事件循环导致前端卡顿；无法运行 PyTorch 模型 |
| **复杂度** | 中 |
| **适用场景** | 仅需轻量级浏览器端推理（如情感分析） |

### Option C（✅ 采纳）：Next.js BFF + 独立 Python FastAPI AI 微服务

| 维度 | 评估 |
|------|------|
| **优点** | 完整 Python AI 生态；计算与业务隔离互不影响；可独立扩缩容；论文算法可直接复用；渐进式引入零风险 |
| **缺点** | 新增一个服务的运维成本；跨服务通信增加网络延迟（内网 < 1ms 可忽略）；需要维护两种语言代码 |
| **复杂度** | 中-高 |
| **适用场景** | 需要深度 AI 能力、私有知识库、论文算法复现 |

### Option D：完全用 Python（Django/FastAPI）重写整个后端

| 维度 | 评估 |
|------|------|
| **优点** | 统一语言栈 |
| **缺点** | 放弃 Next.js 的 SSR/ISR/RSC 生态优势；前端开发体验倒退；大量现有代码需要重写；Prisma 的类型安全查询需要用 SQLAlchemy 替代 |
| **复杂度** | 极高 |
| **适用场景** | 从零开始的纯 API 项目（不适用于本项目） |

---

## Decision

**采纳 Option C：Next.js 作为 BFF + 独立 Python FastAPI 微服务处理 AI 计算。**

但遵循 **渐进式引入** 原则：
- Phase 1 使用 Option A（仅调用第三方 API 验证需求）
- 需求确认后才引入 Python 服务（Phase 2）
- 绝不在需求未验证前过早投入基础设施

---

## Rationale

1. **Python 是 AI 的"母语"** — 所有前沿论文、框架、预训练模型均以 Python 为主
2. **计算隔离** — AI 推理/解析的高 CPU/GPU 消耗不会影响前端用户体验
3. **渐进式风险控制** — 通过分阶段引入，每个阶段都有明确的触发条件和回退方案
4. **架构技能框架验证** — Pattern Selection 明确指出"可以后续抽取服务"而非提前拆分

---

## Trade-offs Accepted

| 我们放弃的 | 为什么可以接受 |
|-----------|---------------|
| 单一语言栈的简洁性 | 个人开发者同时掌握 JS 和 Python；两种语言各司其职 |
| 微服务带来的运维复杂度 | 宝塔面板同时支持 Node 和 Python 项目管理；内网通信简单 |
| 跨服务调用的网络延迟 | 同一台服务器内网调用延迟 < 1ms，可忽略不计 |

---

## Consequences

### Positive
- 获得完整 Python AI 生态
- AI 服务可独立升级、扩缩容、甚至迁移到 GPU 服务器
- 前端业务稳定性不受 AI 计算影响
- 论文算法可直接以 Python 源码形式服务化

### Negative
- 需要维护两种语言的代码和依赖
- CI/CD 流程需要考虑两个服务的协调部署
- 新增服务间通信的错误处理和超时逻辑

### Mitigation
- 建立统一的 API 契约文档（OpenAPI Spec）
- 使用健康检查 + 优雅降级：AI 服务不可用时 Next.js 回退到基础功能
- 通过 Docker Compose 或宝塔面板统一管理两个服务的生命周期

---

## Revisit Trigger

当以下任一条件成立时，需要重新审视此决策：

- Python 服务的复杂度超出个人维护能力 → 考虑全托管 AI 平台（如 Dify, Coze）
- Node.js AI 生态成熟到可以替代 Python（目前不太可能在 2-3 年内发生）
- 团队扩展到多人且主要技能栈为 TypeScript → 考虑统一到 Node.js
