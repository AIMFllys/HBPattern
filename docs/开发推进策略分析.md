## 2. 两种推进策略对比

### 方案 A：先完结前端，再接后端

**优点**：
- ✅ 前端快速达到"可演示"状态
- ✅ 视觉一致性有保障
- ✅ 不会出现前后端冲突
- ✅ Mock 数据层作为合约

**缺点**：
- ❌ 替换 Mock → API 时需要大量重构数据流
- ❌ 筛选、分页、搜索不连真实 API 没意义
- ❌ 3D/地图纯占位没有实际价值
- ❌ 认证方案变化导致重做

### 方案 B：前后端垂直切片并行

**优点**：
- ✅ 每个 Slice 交付"功能完整"
- ✅ 不存在后期大联调
- ✅ 数据流从第一天就是真实的
- ✅ 更早发现架构问题

**缺点**：
- ❌ 起步慢
- ❌ 复杂度高
- ❌ DB 未准备好会阻塞一切

## 3. 推荐策略：混合方案

> **先做"薄前端收尾 + 技术决策"，然后按垂直切片推进**

```
 Phase 0: 收尾+决策 (~1天)     Phase 1~N: 垂直切片推进
 ┌──────────────────────────┐  ┌────────────────────────────┐
 │ ① 修复 4 个 Critical Bug  │  │ Slice 1: 基础层 + 认证      │
 │ ② 抽取 Footer 共享组件    │  │ Slice 2: 画廊完整流         │
 │ ③ 统一认证方案决策         │  │ Slice 3: AI 创作完整流      │
 │ ④ 统一数据库方案决策       │  │ Slice 4: 地图 + 探索完整流  │
 │ ⑤ 移动端汉堡菜单          │  │ ...                        │
 │ ⑥ 补 loading/error 页     │  └────────────────────────────┘
 └──────────────────────────┘
```

### 为什么选混合方案？

1. **Phase 0 是"技术债清理"**，不管后续走哪条路都必须做
2. **技术决策必须先定**，否则后续工作建立在错误基础上
3. **之后按垂直切片**，因为纯 UI 壳体没有实际价值
4. **避免大规模联调**，每个 Slice 完成就是可用功能

## 4. 待定技术决策

### 认证方案

| 维度 | Supabase Auth（现状） | NextAuth.js（文档规划） |
|------|---------------------|----------------------|
| 当前代码量 | ✅ middleware + callback 已实现 | ❌ 零代码 |
| 中国访问 | ⚠️ 海外，可能慢 | ✅ 自建，国内快 |
| 与Supabase | ⚠️ 认证在海外，数据在国内 | ✅ 全部内网 |

**建议**：开发/演示阶段保留 Supabase Auth；部署国内时迁移 NextAuth.js

### 数据库方案

| 维度 | Supabase Postgres | 本地 Prisma Dev | Supabase |
|------|-------------------|----------------|-----------|
| 适合阶段 | 开发/演示 | 纯本地开发 | 生产 |
| 延迟 | 海外 100-200ms | 本地 0ms | 同地域 <1ms |
| 成本 | 免费 500MB | 免费 | 按量 |

**建议**：开发用本地 Prisma Dev → 演示用 Supabase → 生产用Supabase

## 7. 结论

| 问题 | 回答 |
|------|------|
| **先前端还是一起？** | **混合**：1天前端收尾，然后垂直切片前后端一起 |
| **为什么不纯前端先完？** | 筛选/分页/搜索不连 API 没意义，做完还要大改数据流 |
| **为什么不直接并行？** | 有 4 个 Bug + 代码重复 + 2 个未决技术决策，必须先清理 |
| **第一步做什么？** | Phase 0：修 Bug → 抽组件 → 登录页 → 汉堡菜单 → 定方案 |
